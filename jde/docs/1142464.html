F00095 Record/Object Reservation in EnterpriseOne Interactive Application

Purpose of Document

This document provides a list of interactive applications which use the record/object reservation routine to maintain higher integrity for certain transactions.

This document is intended for the EnterpriseOne Developer and system administrator who will be implementing and troubleshooting the F00095 Record/Object Reservation in EnterpriseOne Interactive Applications.

 

Overview

The purpose of Record Reservation is to maintain higher integrity by preventing other users/processes from accessing a certain table simultaneously by recording data into the F00095 (Business Object Reservation table). By implementing this constraint, two different processes do not compete to update/delete a certain table at the same time.

The Record Reservation functionality will typically be implemented for applications which have to update critical information or within an application which updates transactions created by another application. This functionality is implemented by using Business Functions that can be called to reserve a record.

Refer to <Document 1289755.1> - E1: BSFN: Record Reservations Related Business Functions in EnterpriseOne to understand how record reservation related business functions can be implemented.

  • Column Explanation

    • Application: Calling interactive application ID and it's description
    • UDC 00/RR: Whether routine is calling the business function Validate Application Id
    • F00022: Whether routine reads UDC 00/RR based on table name RRESERVE or a specific application ID
    • Reserve by: Business Function which validates and reserves record
    • Generic Key: Key to reserve based on Target Table
    • Apps: Application which is calling Reserve by
    • Name Object (Target): Target table which reserves record
    • Detail Error: Whether a certain application shows a detailed error by calling the business function Set Object Reservation Text Substitution Error
    • Remove by: Business function which deletes a record from F00095

NOTE:
  • Some applications may not have a release reservation routine because it will be released by a following form/sub-form
  • Some application may not follow the general routine in <Document 1289755.1> E1: BSFN: Record Reservation Related Business Functions in EnterpriseOne
  • Some applications may write/delete more than two tables through the record reservation process
  • It is very important when you need to update any tables listed above to minimize locking and blocking
  • Depends on your EnterpriseOne Release the application P43060 may not be in the list of UDC 00/RR. In case you hit any locking against F4209 and F4311, review your exiting UDC and add a record for this entry
  • In some release or demo data P42101 may not be in UDC 00/RR. If you run P42101 add it into UDC 00/RR
  • The way it reserve/release record reservation in P42101 is different from other applications
  • If you have any custom routine which may update tables listed above, do implement record reservation routine
