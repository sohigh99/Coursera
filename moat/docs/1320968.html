Notes:
  • Applications written here are non-exhaustable and more applications may be added in the future.
  • Refer to individual product group or application to better understand the implementation of transaction processing.
  • Transaction Processing is "enabled" at the application level, and cannot be controlled through JDE.INI settings.
  • Transaction Processing is how an application is coded to handle commits and rollbacks.
  • Transaction processing post-commit event processing
    • EnterpriseOne Tools release 8.91 (application release 8.9 and above)
    • Two new events have been created for application developers to better manage data integrity related issues in the event that a transaction fails. The Post Commit Event and Post Button Click – Asynch event can be used in conjunction with JD Edwards EnterpriseOne transaction processing. These events allow for the recovery of operations which were outside of the transaction boundry in the event of a failed transaction.
    • Example of application: P4312 - Purchase Order Receipt
  • Application specific note
    • Transaction Processing flag is removed in P3002 (Bill of Material) to prevent possible deadlock in updating F4102.MRPC and F4102.LLX in a single End Doc routine with different BSFNs
    • Business Services JP43E010 (Mobile Requisition Self Service Approval) and JP430000 (Purchase Order Approval) have transaction processing implemenation from EnterpriseOne application release 9.0 Update 2. Refer to <Document 1392963.1> JD Edwards EnterpriseOne Announces Availability of Mobile Applications for Release 9.0
    • Transaction Processing for the UBEs R41413 (Inventory Cycle Count Update) and R42500 (Batch Shipment Confirmation) are avaliable from EnterprinseOne application release 9.0 Update 1 and upward
    • Transaction Processing is off for Taiwan localication in running P4314 (Voucher Match) to handle GUI application(s).
