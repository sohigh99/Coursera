Detail Information on Text File Processor R007102 (Purge Processed Transactions)

Purpose of Document

This document describes the as-is functionality of R007102 (Purge Processed Transactions). Currently multiple applications may interface Text Processor, example, Bank Tape, JinShui tax report and so on. This UBE is used across many applications.

Refer to the following guide for more detail:

JD Edwards EnterpriseOne Applications Financial Management Fundamentals Implementation Guide
27 Working with the Text File Processor
27.3 Purging a Text Batch in the Text File Processor

Table Layout

The following explains the table relationship between the header and detail tables used by R007102.

    • F007101 - Text Processor Header

Seq Alias Description Explanation
1 GPBT* BatchNumber [8] A number that identifies a group of transactions that the system processes together. The system assigns this number using Next Numbers. NN (00|7)
2 GPTN* TransactionNumber [14]

A unique alphanumeric field that identifies a text batch. The transaction number is determined by the process that creates the text batch.

Commonly the application ID which writes F007101/F007111 is to be written here.

The user ID of the person who created the text batch. You can sort text batches by user ID so that users can select their own batches.

This value can be used when you run submit R007102 from Batch Versions.

Data Structure

  • Report Data Structure

R007102 uses the following report data structure. This is to enable user to call R007102 either through report interconnection or through work with batch versions.

Structure Member Name Data Item Alias Description
mnBatchNumber BatchNumber GPBT This value is to use for Data Selection when R007102 gets called from P007101
szTransactionNumber TransactionNumber GPTN This value is to use for Data Selection when R007102 gets called from P007101
szAdditionalPurgeProcess Additional Purge Process GPPR This parmaeter contains executable UBE ID. If the value is specified then it can be called from R007102
cProcessFlag ProcessFlag GPSP

Ths parameter is to determine,

  • Whether to purge or not (e.g., 2 or 7)
  • Whether it is outbound or inbound

  • Processing Option Data Structure

Processing option below determines whether to purge header file (F007101) or not.

Tab Parameter Help Text
Process 1. Purge Header Records
Blank = Do not delete header records
1 = Delete header records		 Use this processing option to determine whether to delete corresponding header records from the Text Processor Header table (F007101) along with detail records from the Text Processor Detail Table (F007101). Deleting only detail records updates the processed flag in the Text Processor Header file to 2 (purged/outbound) or 7 (purged/inbound). Records marked as purged cannot be reset to an unprocessed status. Valid values are:

  • 1 - Purge header and detail records
  • Blank - Purge only detail records


This parameter determines whether to delete (note that there is no archive option) header file (F007101) or not. In case it is not purged, the field F007101.GPSP is updated with either '2' or '7' depending if it is for outbound or inbound respectively.

Data Selection

  1. Through report interconnection from P007101 (Text File Processor) which can be called from Work With Text Batches (W007101A). Note that version is hard coded with R007102|ZJDE0001. And error 143T () is to be issued when GC Status Code (F007101.GPSP) is either 2 (Purged/Outbound) or 7 (Purged/Inbound). For this case, any runtime data selection will be ignored.
    GC Batch Number -> RI mnBatchNumber
    GC Transaction Number -> RI szTransactionNumber
    GC Additional Purge Process -> RI szAdditionalPurgeProcess
    Data selection is hard coded as below:
      • F007101.GPBT = RI mnBatchNumber = GC Batch Number
      • F007101.GPTN = RI szTransactionNumber = GC Transaction Number
  1. Submit through Work with Batch Versions (P98305W for HTML client)

Runtime data selection but it excludes where F007101.GPSP IN ('2', '7') because detail F007111 has been purged previously.

Detail Routine


Explanation:

  • R007102 is running based on only header (F007101)
  • Regardless the way you call R007102, records where F007101.GPSP '2' or '7' are to be excluded. So in case you decided to purge header file along the way, there is no way to purge these files
  • If Additional Purge Process (F007101.GPPR) is specified for a specific UBE (e.g. R03B673) then this UBE is to be called
  • Nested UBE gets called through business function AdditionalPurgeProcess (B0001001, Call Additional Purge Process) via API jdeLaunchUBEEx ()
  • If nested UBE fails it sends out error workflow message (this is natural because R007102 does not generate any output so work center is proper location to check) with LM0149
    • CAUSE.... UBE &1 (additional purge routine) failed to execute correctly for
      Batch Number &2, Transaction Number &3
      RESOLUTION.... Please contact your system support person to manually complete
      the purge process.
    • &1 is pointer to value specified in F007101.GPPR (for instance, R03B673)
  • In case you do not have any custom purge batch, do not specify this column (F007101.GPPR)
  • It submits only default version ZJDE0001. So in case you have custom routine to create a certain temp file and purge application to purge it, ensure that you have default version ZJDE0001 for the same.
  • Additinal purge application is called by business function AdditionalPurgeProcess (B0001001, Call Additional Purge Process)

Considerations

  • Currently there is no repository to archive purged data
  • It will be good practice that you purge both header/detail at once after Text File is created/copied properly based on F007111 to minimize confusion because multiple application may require to process text file through P007101
