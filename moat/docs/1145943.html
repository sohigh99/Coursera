F00095 Record/Object Reservation in EnterpriseOne Batch Application

Purpose of Document

This document provides a list of batch applications which use the record/object reservation routine to maintain higher integrity for certain transactions.

This document is intended for the EnterpriseOne Developer and system administrator who will be implementing and troubleshooting the F00095 Record/Object Reservation in EnterpriseOne Batch Applications.

Overview

The purpose of Record Reservation is to maintain higher integrity by preventing other users/processes from accessing a certain table simultaneously by recording data into the F00095 - Business Object Reservation table. By implementing this constraint, two different processes do not compete to update/delete a certain table at the same time.

The Record Reservation functionality will typically be implemented for applications which have to update critical information or within an application which updates transactions created by another application. This functionality is implemented by using Business Functions that can be called to reserve a record.

Refer to <Document 1289755.1> - E1: BSFN: Record Reservations Related Business Functions in EnterpriseOne to understand how record reservation related business functions can be implemented.

  • Column Explanation

    • Application: Calling batch application ID and its description
    • UDC 00/RR: Whether routine is calling the business function Validate Application Id
    • F00022: Whether routine reads UDC 00/RR based on table name RRESERVE or a specific application ID
    • Reserve by: Business Function which validates and reserves records
    • Generic Key: Key to reserve based on Target Table
    • Apps: Application which is calling Reserve by
    • Name Object (Target): Target table which reserves record
    • Detail Error: Whether a certain application shows detail error by calling the business function Set Object Reservation Text Substitution Error
    • Remove by: Business function which deletes a record from F00095

NOTE:
  • R47500 is to be called by R42500. R42500 sets F47132.EDOC = F4211.DOCO. So the result of Generic Key between DOCO+DCTO+KCOO and EDOC+DCTO+KCOO are same.
  • New <Bug 14560156>: R47500 RECORD RESERVATION NOT WORK has implemented for EnterpriseOne application release 9.0 and 9.1, which changes Generic Key from EDOC+DCTO+KCOO to DOCO+DCTO+KCOO
  • Some applications may not have the release reservation routine because it will be released by the called UBE
  • If event rule of UBE listed above calls "Validate Application Id", which determine whether to reserve/lock based on UDC 00/RR or not
  • In case reserve and release is same business function, reserve record and release/delete record will be handled by ActionCode
  • UBE R42750 amd R42950 are available only for EnterpriseOne Application release 9.1
  • Make it sure all baseline ESUs are in place for 9.1 release to make full use of these enhancement
  • In case you need to reserve record in your running these application, add value into UDC 00/RR - Reserve Record
